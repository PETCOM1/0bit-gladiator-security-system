Role & Access Manual

Six roles govern Gladiator Pro, split across two tiers: a small platform staff that runs the business, and the tenant-side staff each security company uses to run its own guards. This manual defines what each role is for, who creates whom, and what they can and cannot touch.

03

The six roles

In descending order of reach — platform tier first, then tenant tier.

TIER P1

Super Admin

SUPER_ADMIN

Ultimate authority over the entire platform, every tenant included.

Created by No one — the root account, provisioned directly.
Creates Admin accounts.
Can do
  • See and manage every user on the platform, in any tenant, any role
  • Permanently delete a user account — the one unrecoverable action in the system
  • Invite or remove Admin accounts
  • Platform-wide analytics, audit log, and system settings
Sidebar
DashboardAdminsUsersAnalyticsAudit LogSystem
TIER P2

Admin

ADMIN

Runs day-to-day platform operations — nearly everything a Super Admin can, minus the power to create other Admins or delete accounts outright.

Created by Super Admin only.
Creates Account Manager ("Staff Member") accounts.
Can do
  • Manage tenants, including suspending or reactivating one — an Admin-exclusive power an Account Manager doesn't have
  • Manage subscription plans, helpdesk, and platform-wide analytics
  • Reach directly into a tenant's Sites — create, edit, or remove them
Cannot do Create another Admin, or permanently delete a user account.

TIER P3

Account Manager

ACCOUNT_MANAGER

Brings new security companies onto the platform and manages the relationship afterward. Shown in the product as "Staff Member."

Created by Admin or Super Admin.
Creates Nothing directly — but onboarding a tenant automatically creates that company's first Manager account.
Can do
  • Onboard new tenants and view every tenant on the platform (own onboards flagged for reference)
  • View subscription plans read-only, handle helpdesk tickets
  • Own analytics dashboard tracking their onboarding activity
Cannot do Suspend or reactivate a tenant after onboarding, or create/edit plans.
Sidebar
DashboardTenantsAnalyticsPlansHelpdesk
TIER T1

Manager

MANAGER

Owns one security company's entire account — every site, every guard, company-wide.

Created by Automatically, the moment an Account Manager (or Admin/Super Admin) onboards the company — it becomes that tenant's first user.
Creates Site Manager and Guard accounts, for any site in the company.
Can do
  • See every site and every user in the company, unfiltered
  • Invite staff, and move a user between Site Manager and Guard
  • Company-wide analytics, audit log, settings, and support
Sidebar
DashboardSites & OpsStaff & TeamAnalyticsAudit LogSettingsSupport
TIER T2

Site Manager

SITE_MANAGER

Runs the day-to-day security operation at one specific site.

Created by A Manager, and assigned to one site.
Creates Guard accounts — automatically pinned to their own site, they can't staff anywhere else.
Can do
  • Manage personnel and posts at their site
  • Run operations — shift scheduling, occurrence book, incidents, visitors
  • Site-level analytics: KPIs, attendance, coverage, alerts, and a printable Occurrence Book & Visitor register
Sidebar
DashboardPersonnelOperationsAnalyticsSupport
TIER T3

Guard

GUARD

The security officer physically on duty — the base of the org chart, and the one actually protecting the site.

Created by A Manager or their Site Manager.
Creates Nothing — no invite or user-management rights.
Can do
  • Clock in and out of assigned shifts
  • Log Occurrence Book entries and report incidents
  • Check visitors in and out at their post
  • View their own attendance history and shift roster

Engineering notes

Site Manager invite scope isn't locked server-side. The invite endpoint only blocks inviting Admin/Super Admin — it doesn't stop a Site Manager from requesting the Site Manager role itself. Today's UI never offers that option, but it's a UI-level restriction, not an API one, worth closing.
Admin's reach into tenant Sites is broader than the platform/tenant split suggests. Admin can directly create, edit, or delete a tenant's Sites — the same permission Manager has — rather than staying purely at the oversight level.
